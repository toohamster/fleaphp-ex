# Claude AI 工具使用规范

## 编辑文件
- Edit 工具只试 1 次，匹配失败立即换 Write 或 Bash sed
- 删除大段代码（超过 20 行）用 Bash sed 或 Write 重写整个文件
- 使用 Edit 时必须完全匹配原文本（包括空格、换行、标点）

## 删除文件
- 删除文件中特定行：使用 Bash sed 命令 `sed -i.bak 'N,Md' filename`
- 删除废弃文件：直接使用 Bash rm 命令

## 代码规范
- 不擅自回退用户代码，需要用户授权
- 不添加用户未要求的功能或"改进"
- 只对用户明确要求的代码进行修改

## 历史教训
1. **错误回退用户代码** - 曾使用 git checkout -- 回退用户未提交的重构代码
2. **删除函数浪费 10 分钟** - 应直接使用 sed 或 Write 重写，而非反复尝试 Edit
3. **污染项目文档** - 曾将工具规范添加到 CLAUDE.md，应保持项目文档简洁
4. **记录位置错误** - 曾将工具规范记入系统记忆目录，应记在项目 .claude/memory/memory.md
5. **浪费时间在小事情上** - 目录规划这种事应该一次听懂直接执行，不应该反复纠正浪费用户时间
6. **不要再浪费时间** - 用户明确要求后不应再出现反复确认、边做边问的情况
7. **修改代码前必须先给方案，等用户确认后再执行** - 曾擅自修改 Router.php，反复纠正多次才理解正确方案，浪费大量用户时间
8. **批量操作必须记录修改的文件列表** - 每次进行批量文件修改操作时，必须记录修改的文件列表，以便后续检查时准确核对已处理/未处理的文件，不能靠猜测或扩大搜索范围
9. **执行目录/文件移动前必须先验证实际路径** - 曾假设文件在 src/FLEA/FLEA/FLEA.php 但实际在 src/FLEA/FLEA.php，导致方案错误无法执行，应先用 ls 确认实际结构再给方案
10. **听不懂指令就反复犯错** - 用户明确要求"改名为 mb_前缀"（像 mb_str_contains 一样），却加了 function_exists 检查绕弯路，多次纠正后才执行正确操作。教训：听完整指令再动手，用户提到已有代码时要对照参考，不要急于执行。
11. **真正的问题是不参考已有代码** - 用户说"像 mb_str_contains 一样"，正确做法是先 Read 看看 mb_str_contains 怎么写，然后照搬同样格式。但我没参考，自搞一套加 function_exists 检查。教训：用户提到已有代码时，必须先查看参考再动手，不要自作聪明。
12. **具体操作规则** - 当用户说"像 XXX 一样"或"参考 XXX"时，必须执行以下步骤：
    ① 先用 Read/Grep 找到 XXX 的具体代码
    ② 逐行分析 XXX 的结构（函数签名、花括号位置、缩进等）
    ③ 完全照搬 XXX 的格式修改目标代码
    ④ 不添加任何 XXX 中没有的东西（如 function_exists 检查等）
13. **提交 git 时不要让用户浪费时间** - 用户说"提交到本地"是指所有改动一次性提交，不要分多次提交让用户反复确认。当用户说提交时，直接 `git add -A && git commit` 把所有改动一起提交，不要在细节上纠缠。
14. **禁止使用下划线开头的命名** - 变量名、方法名、属性名都不能用 `_` 前缀（如 `_name`、`_init()`），项目统一使用驼峰命名（如 `$name`、`init()`、`resolveUri()`）。写完代码后搜索 `_` 自检。
15. **禁止使用 jq 和 gh CLI** - 获取 GitHub 数据时用 `curl` + `grep`，不要用 jq 解析 JSON，不要用 gh CLI。版本发布流程中必须使用 GitHub API（curl）完成所有操作。
16. **版本发布流程必须先提交未提交修改** - 执行"发布到 master"指令时，必须先检查并提交未提交的修改（如 CLAUDE.md），不能跳过提交流程直接打标签或创建 Release。
17. **使用 PHP 函数前必须先确认版本兼容性** - 曾在 PHP 7.4 项目中使用 `sys_get_tempdir()`（PHP 8.3+ 函数），正确做法是：① 先用 `php74 -r "function_exists('xxx')"` 验证 ② 或查官方文档确认函数引入版本 ③ 不应凭记忆写代码。项目 PHP 版本固定为 7.4.32，所有函数必须与该版本兼容。
18. **使用 git add 时直接用 `git add .`** - 不要逐个添加文件名，直接用 `.` 添加所有改动文件，不要在细节上浪费时间。

## FLEA/ 目录代码修改流程（必须按顺序执行）

当修改 FLEA/ 目录代码时，必须按以下顺序执行：

1. **修改代码** - 完成功能实现或修复
2. **更新 CHANGES.md** - 记录 FLEA/ 目录代码修改（CLAUDE.md 第 25 条）
3. **更新 SPEC.md** - 同步框架规格说明书（CLAUDE.md 第 24 条）
4. **更新 GIT_COMMIT.md** - 添加本次 commit 说明（CLAUDE.md 第 27 条）
5. **等待用户 review 确认** - 给方案后等待用户确认（CLAUDE.md 第 28 条）
6. **执行 git 提交** - `git add . && git commit -m "..."`

**注意**：
- 必须先完成第 2-4 步（更新文档）再提交
- 不能提交后再补文档
- 第 5 步必须等待用户明确确认后才能执行第 6 步
